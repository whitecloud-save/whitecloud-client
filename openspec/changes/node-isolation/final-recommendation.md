## 实施建议

### 🎯 推荐方案：渐进式迁移

#### 理由

1. **降低风险**: 避免一次性大规模修改
2. **逐步验证**: 每次迁移后可以测试验证
3. **灵活调整**: 可以根据实际情况调整策略
4. **持续交付**: 不影响正常开发进度

#### 实施步骤

**阶段 1: 准备阶段** (已完成 ✅)
- ✅ 搭建基础设施
- ✅ 创建迁移指南
- ✅ 提供示例代码

**阶段 2: 示例迁移** (1-2 天)
- 迁移 1-2 个简单的服务（如 setting.service）
- 验证迁移流程
- 完善迁移指南

**阶段 3: 逐步迁移** (按需进行)
- 优先级 1: 新功能开发时迁移相关代码
- 优先级 2: Bug 修复时迁移相关代码
- 优先级 3: 代码重构时批量迁移

**阶段 4: 最终清理** (待定)
- 当大部分代码迁移完成后
- 统一清理旧代码
- 完全移除 Node 依赖

### 📋 迁移优先级建议

#### P0 - 立即可以迁移
1. **setting.service.ts** - 简单，风险低
2. **新开发的代码** - 直接使用新架构

#### P1 - 优先迁移
3. **save.ts** - 存档功能，影响范围明确
4. **remote-save.ts** - 代码量小，容易迁移

#### P2 - 谨慎迁移
5. **game.service.ts** - 服务层，中等复杂度
6. **oss.service.ts** - 文件上传，中等复杂度

#### P3 - 最后迁移
7. **game.ts** - 核心实体，复杂度高，风险大

### ⚠️ 不建议的做法

1. **一次性全部迁移**
   - 风险太大
   - 难以测试
   - 可能影响稳定性

2. **急于迁移核心实体**
   - game.ts 太复杂
   - 应该等积累足够经验后再迁移

3. **忽略测试**
   - 每次迁移后都要测试
   - 确保功能正常

## 替代方案：适配器模式

如果暂时不想迁移，可以创建适配器层：

```typescript
// src/angular/app/library/node-adapter.ts
import {workerAPI} from './worker-api-instance';

export class NodeAdapter {
  static fs = {
    readFile: (path: string) => workerAPI.fs.readFile(path),
    writeFile: (path: string, data: Buffer | string) =>
      workerAPI.fs.writeFile({path, data}),
    exists: (path: string) => workerAPI.fs.exists(path),
    // ... 其他方法
  };

  static path = {
    join: (...paths: string[]) =>
      workerAPI.path.join({paths}),
    // ... 其他方法
  };
}
```

**优点**:
- 最小化代码修改
- 降低风险
- 可以逐步迁移

**缺点**:
- 增加了一层抽象
- 不是最终目标
